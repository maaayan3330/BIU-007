advanced options to implements for improving the model:

1. Morphological Unpacking (Advanced Preprocessing)
    - The Problem: Hebrew is agglutinative. Prefixes like "and" (ו), "in" (ב), "to" (ל), and "the" (ה) are attached directly to the root word.
    Multilingual BERT's tokenizer is optimized for languages like English. When it sees "והמניאק" (and the maniac), 
    it doesn't recognize the root toxic word because the prefixes change the sub-word tokenization entirely. 
    The model sees a rare string of tokens instead of a known curse word.
    - The Solution: Implement a morphological analyzer before your Phase 1 cleaner. You can use libraries like Trankit, yap (Yet Another Parser), 
    or even basic regex rules to strip standard Hebrew prefixes from nouns.
    Transformation: You programmatically split "והמניאק" into "ו ה מניאק".
    Why it works: This feeds the isolated, naked root word directly into BERT, 
    drastically increasing the chance it recognizes the toxic tokens it was trained on.

2. The Translation Ensemble (Zero-Shot Cross-Lingual Transfer)
    - The Problem: The model was trained heavily on English toxicity and only lightly on Hebrew. 
    Its internal representation of English insults is vastly superior to its Hebrew representation.
    - The Solution: Add a localized, high-speed translation model to your pipeline 
    (like Helsinki-NLP/opus-mt-tc-big-he-en from Hugging Face) and run a dual-inference ensemble.

    When Hebrew text arrives, run it through the toxicity model ->
    In parallel, translate the chunk to English, and run the translation through the toxicity model ->
    Calculate a weighted average of the two scores ->
    the math: Score_final = alpha * score_he + (1-alpha)*score_eng (alpha is hyperparameter set to 0.3 in the start - trusts the English version)

    - Why it works: Even if the translation isn't perfect, curse words and toxic intent translate very well. 
    The model will catch the toxicity in the English translation that it missed in the native Hebrew.

3. Semantic Chunking (Replacing Phase 2)
    - The Problem: Your current sliding window uses a fixed number of words ($N=8$ with overlap $3$). 
    This is computationally blind. It might slice a compound toxic phrase right down the middle, destroying the contextual meaning.
    - The Solution: Move from fixed-length chunking to Semantic Chunking. Use a lightweight NLP library 
    (like spaCy for Hebrew) to parse the sentence structure.
        - Instead of splitting every 8 words, you split the text on dependency parsing boundaries, 
        conjunctions (like "but", "and"), and punctuation.
    - Why it works: This ensures that the text fed into the model contains complete, logically isolated thoughts. 
    The model is forced to judge an exact clause rather than a random slice of words.

4. Parameter-Efficient Fine-Tuning (PEFT / LoRA)
    - The Problem: You have a dataset of false negatives, but you are currently only using it to evaluate the model, not to teach it.
    - The Solution: If you have a GPU available (even a standard gaming GPU or a free Google Colab instance), you can fine-tune the model yourself. 
    Because full fine-tuning is computationally expensive, you should use LoRA (Low-Rank Adaptation).
        - You freeze the base layers of the Multilingual BERT model and inject small, 
        trainable rank-decomposition matrices into the final classification layers.
        - You construct a dataset of the exact false negatives you found, pair them with the label_1 target, 
        and run a short training loop using the Hugging Face peft and transformers libraries.
    - Why it works: You aren't teaching the model Hebrew from scratch; you are just sharply bending its decision boundary to 
    recognize the specific blind spots you found in your dataset.
