1. Data Ingestion Layer (Dataset Loader) - done
Purpose: To read and standardize the evaluation dataset.
Mechanism: Parses your dataset (typically CSV or JSONL) containing the raw comments and their ground-truth labels (e.g., is_toxic: 1 or is_toxic: 0).
Feature: It should shuffle and batch the data to optimize the processing speed and ensure the model is evaluated consistently.

2. Model Execution Wrapper (Batch Inferencer) - done.
Purpose: To run predictions efficiently without spinning up the entire backend.
Mechanism: This component directly imports your classifier's core inference function. It takes the batches from the Data Ingestion Layer and feeds them into the model.
Feature: By stripping away the routing, authentication, and database layers of the backend, this wrapper ensures you are only measuring the raw performance and inference speed of the model itself.

middle step: find a dataset and run it - done
use TextDetox Multilingual Toxicity Dataset - also supports hebrew!

3. Metrics & Scoring Engine (Evaluator) - done
Purpose: To compare the model's predictions against the ground-truth labels.
Mechanism: Calculates the standard classification metrics. For a toxic comment classifier, accuracy is rarely enough due to class imbalances (most comments are usually non-toxic). This engine will calculate:
    - Precision: Out of all comments flagged as toxic, how many actually were?
    - Recall: Out of all truly toxic comments, how many did the model successfully catch?
    - F1-Score: The harmonic mean of precision and recall.
    - Confusion Matrix: A breakdown of True Positives, False Positives, True Negatives, and False Negatives.

4. Artifact & Report Generator (Reporter) - done
Purpose: To log the evaluation so you can compare iterations.
Mechanism: Outputs a summary of the run. It can print a clean table to your console for immediate feedback and save a timestamped JSON or Markdown file (e.g., eval_run_20260410.json) into a local /eval_reports directory.
Feature: Since The Guardian focuses heavily on privacy-preserving security, keeping these reports and datasets strictly local ensures no sensitive testing data is accidentally leaked to third-party logging platforms.
